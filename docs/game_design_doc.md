# 配信アイドル育成ゲーム 仕様書 v1.0

一人の配信者兼アーティストを5年間（ゲーム内時間）育成し、武道館ライブを目指すブラウザゲーム。
TRPG（クトゥルフ）風の判定ロールを全アクションの共通基盤とする。

---

## 1. コアコンセプト

- 単一キャラクターの集中育成（キャラの多さ≠効率）
- 「配信」と「音楽活動」を対等な二軸として絡める
- 放置要素はオフライン時間の擬似進行のみ（メインはアクティブプレイ）
- 最終ゴール：武道館ライブでA評価以上

---

## 2. 時間・行動システム

### 2.1 ターン構造

- 1日 = 1ターン、5年 = 1825ターンで武道館到達を想定
- 1日 **3AP**（行動ポイント）
- 休息以外のアクションはスタミナ（現在値）を消費

### 2.2 スタミナ / メンタル

| リソース       | 減少要因                             | 回復手段                               |
| -------------- | ------------------------------------ | -------------------------------------- |
| スタミナ現在値 | 配信-20 / レッスン-15 / 制作-15 など | 休息（最大値の60%）、毎晩+30           |
| メンタル       | 活動-5、ファンブル事故-15、炎上      | 休息+40、毎晩+10、遊びに行く（大回復） |

- スタミナ最大値はパラメータ「スタミナ」に連動して成長: `スタミナ上限 = 100 + スタミナパラメータ×0.5`（マイルストーン6で確定）
- 現在スタミナ/メンタルが25未満になったら休息推奨（botシミュレーション基準）
- **スタミナ不足時はそのアクションを実行できない（マイルストーン6で確定）。** 各アクションの消費スタミナ（配信-20/レッスン-15/制作-15/アルバイトは種目ごとに個別）に現在スタミナが満たない場合、そのアクションは選択できず、休息（消費なし・回復のみ）を挟む必要がある。実プレイフィードバックを受けて追加。メンタルは消費不足でアクション自体をブロックする仕組みは無く、低下時の判定ペナルティ（下記）のみ
- メンタル低下時は全判定にマイナス修正（例: 30%以下で-10）
- メンタルは常に0〜100固定（スタミナと違い成長パラメータではない）

### 2.3 オフライン擬似進行

- 「おかえりなさいログ」形式：不在中の出来事を復帰時にまとめて報告
- ファン微増減、体力/メンタル自然回復、小さなランダムイベント
- 収益・経験値のバルク付与は行わない（オンライン中心の設計を守る）

### 3.1 成長方法

- **レッスン**：効率的（成功時+2.2、失敗時+0.7、クリティカルで2倍）。費用は現在値に比例（現在値×1.5G）→ 経済的ソフトキャップ
    - 実装上は§5の統一判定エンジン（7点満点・スコア帯）をそのまま流用する。伝説回（8点以上、クリティカル込みの大成功に相当）で2倍、標準成功以上（3点以上）で+2.2、それ未満（不発/失敗/事故）は最低保証+0.7とする（マイルストーン6で確定）
    - **レッスンの要求値は配信/制作と違いファン数に連動せず、ロールごとに「そのロール自身のパラメータの現在値−成功寄りボーナス(10)」を使う。** 対象パラメータだけでなく、支援ロール（スタミナ/メンタル/ラック等）も自分自身の値と比較する。
      経緯: 当初は配信と同じ`requirement(fans)`を使い、ファン数が伸びるほど成功率が床(5%)へ張り付き進行が止まった（バグ）。次に固定値20を試したが初期パラメータ(10)では基準成功率40%にしかならず不十分だった。次に対象パラメータのみ自己参照（要求値=対象パラメータの現在値、基準成功率常に50%）にしたが、これでも対象パラメータだけ特化して伸ばすと、支援ロールが対象パラメータの値を要求値にされたまま置いていかれ、5%floorに張り付いて全体スコアを引き下げる問題が残っていた（特に専用レッスンの無い経験系パラメータ＝メンタル/愛嬌等が支援ロールとして登場したときに顕著。実プレイフィードバックで「メンタルや愛嬌が成長する手段がない」と判明）。
      最終的に**全ロールを自分自身の値に対して自己参照**にし、さらに成功寄りに10だけ要求値を割り引くことで、レベルや役割に関わらず基準成功率がおよそ60%（低〜中レベル帯）になるよう調整した（高レベル帯では要求値の分母が実効値に応じて伸びるため徐々に50%へ収束する＝経済的ソフトキャップの意図を維持）
- **活動経験**：配信/動画/音楽の実行で副次的に上昇（+0.1〜0.2程度に抑える。シミュレーションでは+0.35は強すぎた）。実装値は+0.15
- 経験系パラメータ（アイデア/カリスマ/愛嬌/交渉/メンタル/ラック）は専用レッスンなし、対応アクション成功時に確率で上昇
    - 確率成長ルール（マイルストーン6で確定）：スコア帯が標準成功以上のとき、そのロールがファンブルしていなければ（＝ロール自体の成否は問わない）20%の確率で活動経験値と同量(+0.15)が加算される。配信・レッスンの支援ロール・動画/楽曲制作のいずれでも共通して適用される
    - **【重要】判定基準は「ロール成功時」ではなく「アクション成功時」（＝スコア帯が標準成功以上か）。** そのロール自身の成否では判定しない。専用レッスンを持たないパラメータは、いったんファン数連動の要求値（配信/制作系）に置いていかれるとロール自体は成功しにくくなるが、他の熟練したパラメータがアクション全体を標準成功以上に押し上げてさえいれば、このパラメータにも確率成長のチャンスが来る。ロール成功を条件にすると、置いていかれたパラメータが二度と追いつけない詰み状態になることが実プレイで発覚したため、この基準に修正した

## 4. キャラメイク

- 基礎値：全パラメータ一律 **10**
- 割り振りポイント：**150**（1pt = +1、作成時上限40）
- **ラックのみ割り振り不可**：3d6×2（6〜36）をロールで決定
- プリセット4種（アイドル型 / シンガーソングライター型 / ゲーマー型 / 企画屋型）＋自由配分

## 5. 判定システム（全アクション共通）

### 5.1 統一処理フォーマット

```
1. アクション選択（配信種別、レッスン種別、曲ジャンル…）
2. ロール構成に従って複数回のd100判定
3. スコア集計（7点満点、重み合算）
4. スコア帯 → 結果テーブル → 資金/ファン/経験値/状態変化
```

### 5.2 成功率式（正規化対抗ロール）

```
成功率 = 50 + (自分の値 − 要求値) / max(要求値, 100) × 100   [%]
5〜95にクランプ
```

- 要求値100未満は「差1 = 1%」の線形、以降は比率ベースに自動移行
- どのスケールでも同じ手触り（K=0.5固定線形は終盤に5/95へ振り切れるため廃止）

### 5.3 クリティカル / ファンブル（d100の出目で判定）

- クリティカル：01〜05 ＋ ラック域拡張（+min(10, ラック/50)）→ 重み2倍計上
- ファンブル：97〜00 → 重みマイナス計上、配信系は事故イベント（小炎上等）の可能性

### 5.4 結果テーブル（7点満点、重心は3〜4点）

| スコア                | 判定     | スコア係数 | 追加効果                       |
| --------------------- | -------- | ---------- | ------------------------------ |
| 8+                    | 伝説回   | 3.0        | ファン増×3、話題状態           |
| 7                     | 大成功   | 2.0        | 資金+50%                       |
| 5〜6                  | 好調     | 1.5        | —                              |
| 3〜4                  | 標準成功 | 1.0        | —                              |
| 1〜2                  | 不発     | 0.4        | —                              |
| 0以下                 | 失敗     | 0.15       | ファン微減                     |
| ファンブル込みで3未満 | 事故     | 0.1        | 小炎上：ファン-1%、メンタル-15 |

※ロール成功率50%×4本の期待スコアは3.5。「3〜4=標準」に重心を置くこと（5点以上を成功にすると失敗率5割で破綻する。シミュレーション検証済み）

### 6.6 アルバイト（序盤の資金源）

1AP・固定賃金＋おまけステータス。ファンは増えない。

| バイト         | 賃金 | スタミナ消費 | ステータス        |
| -------------- | ---- | ------------ | ----------------- |
| ライブスタッフ | 150G | 15           | ボーカル表現 +0.3 |
| 倉庫スタッフ   | 180G | 20           | スタミナ +0.3     |
| カフェ店員     | 140G | 15           | 愛嬌 +0.3         |
| カラオケ店員   | 140G | 15           | トーク +0.3       |
| イベント設営   | 160G | 15           | アイデア +0.3     |
| 携帯ショップ店員 | 170G | 20         | 交渉 +0.3         |
| 呼び込みスタッフ | 150G | 15         | カリスマ +0.3     |

- 結果テーブルは賃金×0.8〜1.2の小変動（大成功でボーナス、ファンブルでも減額なし）
- **賃金は固定で成長しない** → 配信収入のインフレにより自然に卒業する（禁止ルール不要）
- ステータス+0.3はレッスン(+2.2)の1/7以下 → ステ目的のバイト周回は常に非効率
- 検証済み: 賃金150Gで1年目末ファン数が約2.5倍に改善（バイト収入→レッスン→配信成功率の間接効果）。300Gでは下積み感が薄れるため不採用
- **判定方式（マイルストーン6で確定・当初計画のマイルストーン9から前倒し実装）**: ロール構成表が仕様書に無いため、ラック1点の単発判定として扱う（7点満点のスコア帯システムは対象外）。要求値はラック自身の現在値（レッスンと同じ自己参照方式）を使い、ファン数と無関係に常に基準成功率50%を維持する。クリティカルで賃金×1.2（wageVariance上限）、通常失敗（非ファンブル）で×0.8（wageVariance下限）、通常成功とファンブルは×1.0（変動なし。ファンブルは「減額なし」の明記通り）
- **実プレイで発覚した経緯**: レッスン費用（現在値×1.5G）を賄うだけの収入源が配信（ファン数依存）しか無く、序盤〜中盤で67.5%のレッスン試行が資金不足のため配信に振り替わり、成長が要求値カーブに追いつけない停滞が発生した。アルバイトによりファン数と無関係な固定収入を確保したことで解消（シミュレーション検証: 365日でファン数78→656の加速成長を確認）
- **携帯ショップ店員・呼び込みスタッフを追加した経緯（マイルストーン6で確定）**: 実装調査の結果、交渉（negotiation）は既存実装内のどのロール構成にも一度も出現せず完全な成長手段ゼロ、カリスマもロックジャンル選択時と動画公開工程の2箇所のみで極めて薄いことが判明した。既存5種と同じ形式（固定賃金＋statGain+0.3無条件加算）でこの2種を追加し、経験系パラメータ全種に最低限の成長手段を確保した

## 8. 経済モデル（シミュレーション検証済み定数）

通貨単位：G

### 8.1 配信

```
同接数     = ファン数 × 10% × スコア係数
資金       = 同接 × 2G（スコア7で×1.5）
ファン増加 = 同接 × 4% × 飽和補正 + 発見ボーナス
発見ボーナス = 標準成功以上で4人 / 未満1.5人（トレンド倍率が乗る）
飽和補正   = 1 / (1 + ファン数 / 1,200,000)
```

### 8.2 動画（日次で全公開動画を処理、公開後120日で打ち切り）

```
日次再生 = バースト + テール
バースト = 公開時ファン数 × 0.4 × サムネスコア × リーチ補正 / 2.5 × exp(-経過日/2.5)
テール   = 現在ファン数 × 0.0015 × 品質スコア × exp(-経過日/45)
リーチ補正 = 1 − exp(-前回投稿からの経過日/5)   ← 視聴者疲労（毎日投稿対策）
収益     = 再生 × 0.5G
新規ファン = 再生 × 5% × 飽和補正
サムネ/品質スコア = 0.8 + 工程スコア/7 × 1.5   （0.8〜2.3）
```

- 週1投稿でリーチ補正75%、毎日投稿で18%（連投は破綻する。検証済み）

### 8.3 楽曲（資産収入）

```
日次DL = 発売スパイク + カタログ底流 + ライブブースト
スパイク = 発売時ファン数 × 6% × 品質 / 7 × exp(-経過日/7)
底流     = 現在ファン数 × 0.00012 × 品質 × 人気度
ライブブースト = 現在ファン数 × 0.8% × ライブ規模 × exp(-ライブ後日数/4)
収益 = DL × 20G
品質 = 1.0 + 制作工程スコア/7   （1.0〜2.0）
```

### 8.4 共通

```
要求値（配信/制作系） = 20 + log10(ファン数/50) × 80
ファン自然減衰 = 0.15% / 日
```

- **係数の変遷**: 元240 →（レッスンが壊れていた時期の検証で）60 →（レッスン/バイト/装備ショップが出揃った状態での再検証で）**80**。
    240は手堅く全パラメータを鍛える戦略でもロール成功率が180日時点で16%程度にしかならず配信がほぼ機能しなかった。60まで下げて解消したが、レッスン・バイト・装備ショップを実装し終えた状態で再シミュレーションすると、今度は60のままだと5年目でも要求値が300程度にしか伸びず、実効値に対して要求値が低すぎて終盤の緊張感が失われる（＝レッスンの種類や配信ジャンルを選ばなくても押し切れる）ことが判明したため80に引き上げた
- **残課題（要再検証）**: 80への再調整は配信＋レッスン＋バイト＋装備のみを使う簡易botで検証したもので、動画/楽曲制作（§8.2/§8.3の追加収入源）を組み込んだ「本来のバランス型bot」では未検証。動画/楽曲を使えばこの簡易botよりファン数は伸びるはずなので、下記5年終了時ファン数200万〜350万という基準は満たせる可能性が高いが、動画/楽曲を含めた回帰テストが整備され次第、再検証すること

### 8.5 検証済み軌道（バランス型bot：動画週1＋曲2ヶ月に1本＋配信）※要再検証（要求値係数変更前の数値）

| 時期    | ファン数    | 収入内訳                   |
| ------- | ----------- | -------------------------- |
| 1年目末 | 約3,000     |                            |
| 2年目末 | 約50,000    |                            |
| 3年目末 | 約40万      | 累計で                     |
| 5年目末 | **約280万** | 配信77% / 動画6% / 楽曲18% |

- 配信のみ：231万人（成立するが次善）
- 動画毎日投稿：53万人（疲労補正により破綻＝退化戦略は封鎖済み）

## 9. トレンドシステム（二層構造）

### 9.1 トレンドオブジェクト

```
トレンド = { 対象タイトル（架空生成）, ジャンル, 規模(S/M/L), 開始日 }
強度(t) = ピーク × (1 − exp(-t/立ち上がり)) × exp(-max(0, t−立ち上がり×2)/半減期)
```

| 規模          | ピーク | 立ち上がり | 半減期 | 発生間隔 | 価格目安 |
| ------------- | ------ | ---------- | ------ | -------- | -------- |
| L（大型新作） | 1.0    | 4日        | 28日   | 約60日   | 8,000G   |
| M（中堅）     | 0.6    | 3日        | 14日   | 約25日   | 4,000G   |
| S（瞬間風速） | 0.35   | 1.5日      | 5日    | 約10日   | 1,500G   |

- 常時3〜5個のトレンドが並存（年間約55タイトル）
- L級のみ発売予告あり、M/Sは突発

### 9.2 効果

| 一致レベル                                | 要求値低下 | 発見ボーナス倍率  |
| ----------------------------------------- | ---------- | ----------------- |
| タイトル一致（当該ゲーム配信/カバー）     | −40 × 強度 | ×(1 + 2.0 × 強度) |
| ジャンル一致（同ジャンル別ゲーム/自作曲） | −15 × 強度 | ×(1 + 0.7 × 強度) |

- ゲーム配信には該当タイトルの購入が必要（投機判断）
- カバー曲：配信は伸びるがDL収入なし。自作曲：ジャンル一致どまりだがDLに繋がる

### 9.3 SNSチェック（情報の鮮度システム）

- 1APでトレンド情報のスナップショットを更新
- プレイヤーに見えるのは**最終チェック時点**の情報（「3日前の情報」表示）
- 表示内容：タイトル / ジャンル / 強度 / 矢印（↑→↓）、L級発売予告、たまに噂（不確実情報）
- おまけロール（ラックのみ使用）：クリティカルでファンアート発見（メンタル回復）、ファンブルでアンチ発見（メンタル微ダメージ）

## 12. 装備システム（ショップ＋アップグレード）

ガチャではなくショップ購入＋レベル強化方式。継続的な資金シンク。

**実装状況（マイルストーン6で前倒し実装。本来はマイルストーン9予定）**: 実プレイフィードバックで「後半は配信収入がレッスン費用を圧倒し、レッスンする意味が薄れる」と指摘され、原因を調査した結果、配信収入はファン数に線形比例するのに対しレッスン費用は仕様書が意図的に対数スケールに抑えたパラメータ値に比例するため、5年目には比率305倍まで乖離することが判明した。この構造自体を変えるのではなく、本来の後半の金主である装備ショップを実装することで解決した。

- マイク・カメラ/照明・PC/ソフト・練習環境は実装済み（判定の実効値または成長量に反映される）
- **衣装（outfit）は購入・強化のみ実装し、効果（ライブ全セクション＋装着ジャンル適性ロール）はライブ（マイルストーン8）実装時にまとめて有効化する**（残課題）。金主としての役割は購入コストのみで十分果たせるため、効果無しでの先行実装とした
- **PC/ソフトの「動画テール減衰延長」効果は未実装**（残課題）。仕様書に数式が無く新定数の発明が必要なため、数式が確定してから実装する
- PC/ソフトの「作曲系ロール」は作曲(composition)工程のみを対象とし、作詞(lyrics)工程は対象外とした（仕様書内で作詞/作曲は一貫して別工程として扱われているため）
- 装備の適用は「スロット→アクション/工程カテゴリ」という仕様書の書き方に忠実に、該当アクションの全ロールに一律加算する方式で実装した（特定パラメータだけに絞らない）
- 装備の購入・強化はAPを消費しないメニュー操作として扱う

### 12.1 スロット（5枠）

| スロット    | 補正対象                                   |
| ----------- | ------------------------------------------ |
| マイク      | 配信系全ロール                             |
| カメラ/照明 | 動画撮影・サムネ工程                       |
| PC/ソフト   | 編集・作曲系ロール、動画テール減衰延長     |
| 衣装        | ライブ全セクション＋装着ジャンル適性ロール |
| 練習環境    | レッスン上昇量%ブースト                    |

### 12.2 強化式

```
補正値   = レベル × 3（レベル上限なし）
強化費用 = 基礎価格200G × 1.35^レベル
```

- 費用は幾何・効果は線形 → 上げ止めどきの費用対効果判断が常に発生
- 5枠Lv40で総費用約1億G（累計収入の2割強）を想定

### 12.3 実効値の定義（重要）

```
実効値 = パラメータ素値 + 装備補正
成功率 = 50 + (実効値 − 要求値) / max(要求値, 100) × 100
```

- §3.2の成長スケール表および全要求値は**実効値ベース**（装備前提）
- 武道館期の実効1150のうち装備が100〜150を担う想定
- 装備を怠る＝要求値カーブから脱落する設計（装備は進行の前提）

## 13. サブコンテンツ：キャラクター選択とデート

メイン効率に一切影響しない装飾系コンテンツ。**好感度→メイン効率への逆流は禁止**（コラボ交渉等も好感度と無関係に既存ロールで処理）。

### 13.1 プレイアブルキャラクター選択

```
ゲーム開始 → キャラクター選択 → ステータス割り振り（§4、変更なし） → 開始
```

- 当面は選択可能キャラ1名。データ構造は複数前提で組む
- キャラごとに専用NPCロースター5名が固定で紐づく

### 13.2 NPC（キャラごとに5名）

二層データ構造:
| 層 | 内容 | 決まり方 |
|---|---|---|
| 固定 | 名前・立場・立ち絵・解禁条件・誕生日 | キャラ定義 |
| 抽選 | 好きなデート先・好きな話題・地雷話題 | **セーブ作成時に rngSeed から抽選** |

- 抽選は人格タグ付きプール（NPCあたり候補8〜10件）から重み付きで3件程度採用
- 周回ごとに好みが変わる → 開発者自身も探索を楽しめる
- 解禁条件はメイン進行と緩く紐づけてよい（例: ライブハウス公演後、外注3回目）。ただし一方向のみ

### 13.3 デート（休息のバリアント）

- 効果: 通常の休息と**同等の回復** + デート代（デート先により100〜300G） + 好感度増減
- フロー: NPC選択 → デート先選択 → 話題選択2〜3回 → 結果
- ロール: 愛嬌2 / トーク2 / ラック1（5点満点・専用結果テーブル、賃金変動なし相当の軽い扱い）
- ミニゲーム性 = **相手の好みの記憶**: 会話中に好みが開示され、以後その知識で選択すると好感度が大きく動く。地雷話題で減少
- 放置スケジュール中はデート不可（デートは手動専用）

### 13.4 報酬（すべて非効率系、2種に限定）

1. **日付イベント**: ゲーム内カレンダー（開始日=4月1日、day % 365）基準。クリスマス（12/24-25）・バレンタイン（2/14）・NPC誕生日（固定日）に専用会話＋好感度ボーナス
2. **好感度カットイン**: 好感度 25 / 50 / 75 / 100 到達で専用カットイン画像＋短文エピソード。アルバム画面から再閲覧可

好感度は 0〜100。エピソードツリー・ホーム画面演出等の重い装飾は実装しない（スコープ外）。

## 14. 実装メモ

- スタック：React / TypeScript（ブラウザ、既存プロジェクトの知見を流用）

    - 楽曲の人気度（popularity）をトレンドと連動させる仕組み（マイルストーン7で対応）
- マイルストーン6で確定した項目（旧・残課題）：スタミナ上限の成長式（§2.2）、経験系パラメータの確率成長ルール（§3.1）、レッスンのスコア帯別成長量（§3.1）、動画/楽曲のジャンル差し込み方式（§6.2/6.3）、動画3工程のスコア→thumbnail/qualityScore変換（§6.2）、楽曲品質スコアの算出（§6.3）
