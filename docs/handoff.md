**⚠️ 進行速度・後半バランスに関する重要な修正履歴**:

実プレイフィードバックを受けて何度も反復修正した。今後同種の数値をいじる際は必ず本セクションを読むこと。

1. **レッスンの要求値バグ（配信と同じrequirement(fans)を誤用）**: ファン数が数百人を超えると配信もレッスンも同時に成功率が床(5%)に張り付く「共倒れ」で進行停止。固定値20を試すも初期パラメータ(10)では基準成功率40%にしかならず不十分だった。
2. **経験系パラメータ（アイデア/カリスマ/愛嬌/交渉/メンタル/ラック）の成長条件バグ**: `experienceParamGains()`が「そのロール自身が成功したか」を条件にしていたため、専用レッスンを持たないこれらのパラメータは、いったんrequirement(fans)に置いていかれると自力では二度と追いつけない詰み状態になっていた（"ほぼクリティカル前提"という体感の原因）。判定基準を「ロール成功時」から「**アクション成功時**（スコア帯が標準成功以上か、ファンブルでなければ対象）」に修正。`growth.ts`参照。
3. **アルバイト未実装による資金不足**: レッスン費用（現在値×1.5G）を賄う手段が配信収入（ファン数依存）しか無く、序盤〜中盤で67.5%のレッスン試行が資金不足のため配信に振り替わっていた。§6.6のアルバイト（固定賃金、ファン数と無関係）を前倒し実装して解消。
4. **requirementLogSlope=240が急すぎた**: 全パラメータを手堅く均等に鍛えるbot戦略でシミュレーションしても、requirementLogSlope=240では180日時点のロール成功率が16%程度にしかならず、§5.4の「50%×4本→期待スコア3.5=標準成功」という設計前提を大きく下回っていた。240→60に引き下げ。
5. **レッスンの要求値、対象パラメータのみ自己参照だと支援ロールが置いていかれる問題**: 4までの修正でレッスン自体は「対象パラメータ自身の現在値」を要求値にしていたが、これは対象パラメータの1ロールにしか適用されておらず、支援ロール（スタミナ/メンタル/ラック等）は依然として**対象パラメータの値**と比較されていた。対象パラメータだけ特化して伸ばすと支援ロールが要求値割れで5%floorに張り付き、「失敗ばかりで成長が遅い」「メンタルや愛嬌が育たない」につながった。`resolveRolls()`の要求値引数を「固定値 or ロールごとの関数」に一般化し、レッスンでは**全ロールが自分自身のパラメータの現在値**を要求値にするよう修正（`judge.ts`, `lessons.ts`）。さらに成功寄りに`lessonRequirementBonus`(10)だけ要求値を割り引き、低〜中レベル帯で基準成功率約60%になるよう調整した。
6. **【今回】後半バランス崩壊（4点セットの指摘）**: 5まで直した状態でも「①後半は配信収入がレッスン費用を圧倒し意味が薄れる、②レッスンしなくても配信だけで十分伸びる、③配信の種類・ジャンルに意味がない、④愛嬌・カリスマの成長手段が無くEDM系歌枠配信だけが最適解」という指摘を受けた。3エージェントで調査した結果:
    - **①は`requirementLogSlope`ではなく構造的な問題**と判明（配信収入=fans線形比例、レッスン費用=param比例だがparamは仕様書が意図的に対数スケールに抑制。5年目で比率305倍まで乖離）。**§12装備ショップを実装して後半の本来の金主を作ることで解決**（本来マイルストーン9予定を前倒し）
    - **②③は期待値計算では逆で、レッスンは配信連打よりAP当たり8〜24倍効率が良い**ことが判明。ただし副次的に`requirementLogSlope=60`だと5年目末（ファン350万）でも要求値は最大310程度にしかならず、要求値が低すぎて終盤の緊張感が失われていた
    - **④は交渉（negotiation）が実装内のどのロールにも一度も出現しない完全な死にパラメータ**であることが判明（カリスマも2箇所のみで薄い）。**携帯ショップ店員（交渉）・呼び込みスタッフ（カリスマ）の2バイトを追加**して解決
    - **requirementLogSlopeを60→80に再調整**。シミュレーションで60/70/80/100/200を比較し、CLAUDE.mdの5年終了時ファン数200万〜350万という回帰基準とのトレードオフを見ながら80に決定（詳細は本ファイル§3参照）

**この種の変更をする際は、必ずbotシミュレーション（一時テストファイル`src/engine/_sim.test.ts`を都度作成・削除、`npm run test -- --run --reporter=verbose <file>`で実行）で実数値を確認すること。体感の報告だけで数値を当てずっぽうに変えない。** 詳細は§7参照。

## 7. デバッグ用ボットシミュレーションの作り方（Tips）

進行ペースの体感がおかしいと感じたら、推測で数値をいじる前に一時的なシミュレーションスクリプトで実数値を確認すること。

```ts
// src/engine/_sim.test.ts（一時ファイル。確認が終わったら削除する）
import { describe, it } from "vitest";
// 各engine関数を直接importして、for文でday/AP消費をシミュレートし、
// 主要な変数（fans/money/params/requirement）をconsole.logする
```

`npm run test -- --run --reporter=verbose src/engine/_sim.test.ts` で実行するとconsole.log出力が見える（`--reporter=verbose` を付けないとstdoutが表示されないので注意）。`npx vitest run`は環境によって`Cannot read properties of undefined (reading 'config')`と一時的に失敗することがあるので、その場合は`npm run test --`経由で実行するか再試行する。

`rng.ts`は`d100()`と`random()`が同じ内部状態を共有しているため、呼び出し回数が変わると以降の乱数列が全部変わる。before/after比較をする際はこの点に注意（単純な数値比較は難しく、傾向で見るしかない）。

requirementLogSlopeのような定数を再調整する際は、`balance.ts`の値を一時的に書き換えて候補値ごとにシミュレーションを回し、結果を比較してから最終値を確定させる（本ファイル§3のテーブルのように記録に残すと後で経緯が追える）。
