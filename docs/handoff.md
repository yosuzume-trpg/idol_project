## 3. 仕様書に無く、実装しながら決めた項目（§14残課題）

**重要: 数値的な判断が伴うので、続きを実装するときはここを必ず読むこと。** 変更する場合は `balance.ts` と `docs/game_design_doc.md` 両方を更新する。

1. **メンタル固定上限100 / 初期スタミナ上限100**（`BALANCE.mentalMax`, `BALANCE.staminaMaxBase`）
   メンタルは成長パラメータではなく常に0〜100固定という解釈で実装。スタミナ上限はパラメータ「スタミナ」との連動成長式が仕様書未確定のため、**今は固定値のまま伸びない**（レッスンでstaminaParamを鍛えても最大値は増えない）。連動式を決めたらここを直す。

2. **§2.2の消費値をbalance.tsに正式追加**
   `activityStaminaCost: { stream:20, lesson:15, production:15 }`、`activityMentalCost: 5`。仕様書の表には書いてあったが定数化されていなかったので今回追加した（仕様書側の数値そのものは変更していない）。

3. **失敗（fail）判定時の「ファン微減」の具体量**（`streaming.ts`の`computeStreamResult`）
   仕様書には「ファン微減」としか書かれておらず量が未確定。暫定的に「発見ボーナス(低)=1.5を符号反転」を採用（`viewerFans - discoveryBonusLow`）。ファン数が大きいと同接由来のviewerFans項が上回ってプラスに転じることもある。要再検証。

4. **活動経験値の対象パラメータ**（`growth.ts`の`LESSON_BACKED_PARAMS`）
   §3.1「経験系パラメータ（アイデア/カリスマ/愛嬌/交渉/メンタル/ラック）は専用レッスンなし、確率で上昇」を踏まえ、**専用レッスンがあるパラメータのみ**、ロール成功時に固定+0.15（`BALANCE.activityGain`）が入る実装にした。経験系パラメータの確率成長は具体的な確率が仕様書に無いため**未実装（常に0成長）**。これがマイルストーン6着手時の要検討事項の一つ。

5. **RNGシードの日次リシード方式**（`gameStore.ts`の`performStream`）
   `createRng(rngSeed + day*1000 + actionIndex)` で、日ごと・その日のアクション順ごとに再シードしている（内部状態を永続化していないため）。ブラウザリロードをまたいでも同じ日の同じ順番のアクションなら同じ出目になる。ただしこれは実装上の選択であり、仕様書に明記された方式ではない。

6. **キャラメイク未実装のため初期パラメータは全て10固定、ラックのみ21**（`gameStore.ts`の`INITIAL_PARAMS`）
   §4のキャラメイク（150ポイント割り振り・プリセット・ラックのダイスロール）はまだ無い。ラック21は3d6×2の期待値を仮採用。キャラメイク画面を作ったらここを差し替える。

7. **配信アクションのうち実装済みは雑談配信・歌枠配信・休息のみ**
   ゲーム配信（ゲーム購入前提）、企画配信（資金コスト額が仕様書で未確定）、コラボ配信（交渉前提）は前提システム未実装のため意図的に見送った。

## 4. 直近確認した既知の挙動（バグではない）

**「500日クリックしてもファン数が70〜80をうろうろするだけで増えない」** という現象を確認済みだが、これは想定内。

原因: `requirement(fans) = 20 + log10(fans/50)×240` は対数的に伸び続けるのに対し、レッスン未実装のため実効値がほぼ10のまま（活動経験値でトーク等が微増するのみ）。ファン数が45人を超えたあたりで要求値が実効値を上回り、以降は成功率が下限5%に張り付く。5%張り付き状態だと「失敗」（fanDelta約-1.5）と「不発」（約+1.6）がだいたい同じ大きさで打ち消し合い、自然減衰（-0.1/日程度）もあって特定の範囲を行き来する均衡状態になる。数式的に一貫した挙動であり、**マイルストーン6でレッスンによる実効値の底上げ手段が入れば解消される**。
